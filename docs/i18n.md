# i18n

## What / Why

`@tknf/oven/i18n` is a thin layer connecting Hono's built-in
`languageDetector` (`hono/language`) with message catalogs. Its scope is
intentionally narrow: it covers messages *the framework itself* emits
(error pages today; validation copy elsewhere in oven), not the
foundation for localizing an app's entire UI (screen labels, body copy,
etc.). If you want a larger catalog for your own app, the `Translator`
class this module exports is designed to be reused directly for that.

The core class is `Translator<C extends Catalog>`: it wraps a
`CatalogBundle<C>` (a language code → catalog map) and a required
`fallbackLanguage`, and exposes a single `t(c, key, params?)` function.
Language resolution reads `c.get("language")` — the value
`languageDetector` middleware sets on the context — falling back to
`fallbackLanguage` whenever that value is absent or not in the bundle.
Because `t` never throws (a missing key falls back to the fallback
catalog, then to the raw key string), a partially-translated catalog
degrades gracefully instead of breaking a response.

## Minimal example

```ts
// src/lib/i18n.ts
import type { CatalogBundle } from "@tknf/oven/i18n";
import { Translator } from "@tknf/oven/i18n";

const greetingBundle = {
  en: { "greeting.hello": "Hello, {name}" },
  ja: { "greeting.hello": "こんにちは、{name}さん" },
} satisfies CatalogBundle<{ "greeting.hello": string }>;

export const { t } = new Translator(greetingBundle, { fallbackLanguage: "en" });
```

```ts
// src/main.ts
import { Hono } from "hono";
import { languageDetector } from "hono/language";
import { t } from "./lib/i18n.js";

const app = new Hono();

// Wire language detection before any handler that calls `t`.
app.use(languageDetector({ supportedLanguages: ["en", "ja"], fallbackLanguage: "en" }));

app.get("/", (c) => c.text(t(c, "greeting.hello", { name: "Taro" })));
```

## Common tasks

### Defining your own catalog and translator

A catalog is a flat `Record<string, string | PluralForms>`; a bundle maps
language codes to catalogs. `Translator`'s constructor throws immediately
if `fallbackLanguage` has no entry in `bundle`, so a misconfiguration is
caught at startup rather than at first request:

```ts
import type { Catalog, CatalogBundle } from "@tknf/oven/i18n";
import { Translator } from "@tknf/oven/i18n";

const en = {
  "cart.itemCount": { zero: "Your cart is empty", one: "{count} item", other: "{count} items" },
} satisfies Catalog;

const bundle: CatalogBundle<typeof en> = { en };
const { t } = new Translator(bundle, { fallbackLanguage: "en" });
```

`{name}`-style placeholders in a template are replaced from `params`; a
placeholder with no matching `params` entry is left untouched. Plural
keys resolve via `Intl.PluralRules(language).select(params.count)`, with
`zero` checked explicitly first when `count === 0` (useful for languages,
like English or Japanese, that have no distinct CLDR `zero` category but
still want dedicated "empty" copy). `other` is required in every
`PluralForms` value — it's the final fallback for any category.

### Wiring `languageDetector` (`hono/language`)

`Translator#t` only *reads* `c.get("language")` — it never detects a
language itself. Apply Hono's `languageDetector` middleware upstream of
any route that calls `t`, so the value is actually present on the
context:

```ts
import { languageDetector } from "hono/language";

app.use(
  languageDetector({
    supportedLanguages: ["en", "ja", "fr"],
    fallbackLanguage: "en",
    order: ["querystring", "cookie", "header"],
  }),
);
```

`supportedLanguages` and `fallbackLanguage` here are `languageDetector`'s
own options (they decide what `c.get("language")` can be set to), which
is a separate concern from `Translator`'s own `fallbackLanguage` (which
decides what catalog `t` falls back to when the detected language isn't
in *your* bundle). The two don't have to list the same languages.

### Translating in a handler or view

Once a language is detected, call `t(c, key, params)` anywhere a
`Context` is available — inside a handler, or passed down into a JSX
view:

```tsx
app.get("/cart", (c) => c.render(<CartPage t={(key, params) => t(c, key, params)} />));
```

### Localizing uploaded-file validation errors

`@tknf/oven/form`'s `validateUploadedFile` returns an English default
message on failure (`failure.message`). To render it in the detected
language instead, use `localizeUploadedFileError` from
`@tknf/oven/form`, which is built on this module's `Translator` (see
[Forms](./forms.md)):

```ts
import { validateUploadedFile } from "@tknf/oven/form";
import { localizeUploadedFileError } from "@tknf/oven/form";

const validation = validateUploadedFile(body.cover, {
  maxSizeBytes: 5 * 1024 * 1024,
  allowedMimeTypes: ["image/png", "image/jpeg"],
});

if (!validation.ok) {
  const message = localizeUploadedFileError(c, validation);
  // message is localized from c.get("language"), falling back to English.
}
```

Its built-in catalog bundles English, Japanese, Simplified Chinese,
Spanish, French, German, Brazilian Portuguese, Russian, Korean, and
Arabic, matched against two-letter base language codes (the same shape
`languageDetector` produces).

## Gotchas / Security notes

- **Without `languageDetector` applied, `t` still works — it just always
  resolves to `fallbackLanguage`.** `c.get("language")` type-checks as
  `string` everywhere (a global `hono/language` type augmentation), but at
  runtime it's `undefined` unless the middleware actually ran on that
  request. `t` guards this with a runtime check, so an app that forgets
  to wire `languageDetector` fails safe rather than throwing.
- **`fallbackLanguage` must exist in `bundle`.** This is checked in the
  `Translator` constructor and throws immediately if violated — treat it
  as a startup-time configuration error, not a runtime one.
- **`other` is mandatory in every `PluralForms` value.** It's the last
  resort both when `Intl.PluralRules` returns a category your catalog
  doesn't define, and when `params.count` isn't a number at all.
- **Translated text is not HTML-escaped by `t`.** Catalog text itself is
  a trusted, developer-owned value, so this is usually a non-issue when
  rendered as a JSX text node (which auto-escapes). If you interpolate
  *untrusted* values into `params` and embed the result into a raw HTML
  string yourself (not via JSX), escape it at that embedding site.
